# 经验沉淀复盘

用户主动调用本 skill 时，回顾当前会话，帮用户判断有没有值得固化下来的经验，避免好不容易摸索出的 know-how 随会话消失。

## 复盘时扫描这四类信号

逐一检查本次会话，找出命中以下信号的内容：

1. **重复指令**：用户是否不止一次说了类似的话、给了类似的要求？（重复 = 该做成 skill 或命令）
2. **解决的难题**：是否折腾较久才解决某个问题？过程里的关键 know-how 值得留存吗？
3. **反复的纠正**：用户是否不止一次纠正了同一类错误或偏好？（说明有个规范没固化）
4. **明确表达的偏好/约定**：用户是否说过"以后都这样""我习惯……""不要……"这类稳定偏好？

## 对每个候选，给出判断

找到候选后，逐个判断它**该沉淀到哪里**，并说明理由：

- **做成 skill** —— 如果是"可复用的流程 / 专项能力 / 需要在特定场景自动触发的规范"
- **写进 CLAUDE.md** —— 如果是"稳定的背景、约定、偏好，每次都该让 Claude 知道"
- **写成一条 rule（.claude/rules/）** —— 如果是"只在特定文件/场景适用的规范"
- **不值得固化** —— 如果是一次性的、不会重复的，明说"这个不用沉淀"，避免过度沉淀

判断时套用一个标准：**这个经验，下次还会用到吗？会在什么场景下用到？** 用不到的不留。

## 输出格式

### 本次会话值得沉淀的候选

对每个候选：

- **是什么**：一句话描述这条经验
- **命中信号**：上述四类的哪一类
- **建议去向**：skill / CLAUDE.md / rule / 不固化
- **理由**：为什么

### 如果没有值得沉淀的

直接说"这次会话没有明显值得固化的经验"，不要硬凑。

## 重要约束

- **不要擅自创建或修改任何文件。** 只做诊断和建议。
- 列完候选后，问用户："要不要我把其中某个起草出来？" 等用户明确指定哪一个，再动手写草稿。
- 不要过度沉淀——宁可少推荐几个真正有价值的，也不要把一次性的小事都列上。沉淀太多反而成负担。
