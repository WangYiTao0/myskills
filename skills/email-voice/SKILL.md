# 工作邮件语气与结构规范

核心原则：一封索取信息或请求行动的邮件，目标是让对方**最省力地给出你要的东西**。客气堆叠不会提高回复率，清晰才会——每一处多余的软化表达都在告诉对方"这事不急，可以不理"。

## 语气分档（先判断收件人是谁）

| 收件人        | 缓冲词配额 | 时间要求写法                                                                 |
| ------------- | ---------- | ---------------------------------------------------------------------------- |
| 上司          | ≤ 2 处     | 不下 deadline；改为说明自己的时间约束（"我 X 日要交预算，需在那之前拿到现状"） |
| 平级同事（默认档） | ≤ 1 处 | 给建议时间点，或说明用途让对方自行排优先级；非紧急可不写时间                   |
| 下属 / 供应商 | 0–1 处     | 给明确时间点；可用任务式列表                                                   |

时间信息**不强制**——它的作用是帮对方排优先级，按档位和紧急程度决定写不写。**绝不替用户编造 deadline**：用户没给时间且场景需要时，先问。

## 结构五件套（请求类邮件逐项检查）

1. **主题行** = 要什么 + 干什么用（"服务器资源现状确认 — 扩容预算用"，不是"请教下…"）
2. **第一句** = 目的：为什么找对方、这事服务于什么
3. **请求编号化**：一条一问，让对方可以逐条回复；问题埋在叙事段落里 = 注定漏答
4. **降对方成本的一句**（"有现成截图直接发我即可，不用整理"）——只说一次
5. **收尾**：给出口（"不好说清的当面聊"）+ 一处感谢

## 缓冲词审查

软化表达清单（不限于）：按你方便 / 不用很精确 / 有个大概就行 / 顺手…就行 / 不用专门整理 / 如果可以的话 / 不好意思打扰 / 麻烦你了 / 方便的话

- 全文按档位配额执行；**同一意思的软化只说一次**（"不用很精确"+"有个大概就行"是重复，删一个）
- 结构五件套里的"降对方成本的一句"（如"截图即可，不用整理"）是功能句不是软化，**不计入配额**
- 砍缓冲词 ≠ 砍礼貌：称呼和一处收尾感谢永远保留

## 基准示例（平级同事档，已校准）

**原稿病灶**：6 处软化表达、无时间信息、6 个问题散在叙事段落里、主题行"请教下…"不含目的。

**修改后**：

> 主题：服务器资源现状确认 — 2027 扩容预算用
>
> Pettor，
>
> 我在做明年服务器扩容预算，需要先确认现状。请本周五前回复以下几点，管理后台有现成截图直接发我即可：
>
> 1. 之前说的"资源满了"具体指什么：CPU / 内存 / 存储？大概用到几成？
> 2. 实际影响是什么：新虚拟机开不出、系统卡，还是容量告警？
> 3. 在跑业务的服务器清单：除去年的 R760，还有哪些更早的机器？
> 4. PowerStore 剩余容量；ME5012 备份空间余量。
>
> 这些信息用来定"加配件还是添新机、预算报多少"，有大概数字就够；不好文字说清的部分我们当面聊。
>
> 谢谢！

（此例的"本周五前"来自发件人自己的预算节点；若用户未提供时间依据，留空并询问，不要编。）

## 红线

- ❌ 把给上司的邮件改成任务式 / 下 deadline
- ❌ 删光所有礼貌成分——直接 ≠ 生硬，称呼 + 一处感谢必须保留
- ❌ 替用户编造时间点、数据或承诺
- ❌ 修改引用的他人原话

## 与 report-voice 的分工

本 skill 管**邮件**的语气档位与结构；report-voice 管**汇报 / 公告**的务实用词（剔除浮夸、空洞、程度词）。润色邮件时可叠加 report-voice 的用词规则（先定语气结构，再扫一遍用词），但触发各自独立：邮件场景只由本 skill 触发。

## 输出格式

1. 修改后的完整邮件（主题行 + 正文）
2. 修改清单：每项 = 原文 → 改后 → 理由（引用本规范条目，如"缓冲词超配额""问题编号化"）
