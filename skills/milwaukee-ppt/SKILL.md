# Milwaukee Tool 工作汇报格式规范

为 Milwaukee Tool 内部汇报类 PPT 定义**统一格式**。
本 skill 只定义"写什么 / 怎么布局"，不负责生成 PPT 文件。

---

## ⚠️ 适用范围：两层规则，先判断走哪条

本 skill 的规则分两类，动手前先判断当前任务属于哪种：

### A. 始终生效（所有 Milwaukee 汇报 PPT 都要守）

- **标题三层结构**（§4）：主标题（话题）+ 副标题（action title）+ 正文 lead（结论 + 数据）
- **每节一个核心信息**（§7）：一节只回答一个问题，超出就拆
- **页面布局规则**（§5：Cover + Body 布局，banner / logo / 页脚）
- **设计系统**（§6：品牌锚点、色板、字体阶梯、红色语义）
- **内容硬约束 / 禁用元素**（§7）

### B. 仅"项目汇报"类生效（满足任一条件才套用）

触发条件（满足任意一条）：

1. 任务明确是**项目汇报 / 项目总结 / 项目月报 / 阶段性项目汇报**；
2. 用户**明确要求**使用 Agenda 结构或 PDCA 标注。

满足才套用下列三条，否则**跳过**：

1. **固定 Agenda 顺序**（§1）：8 节按规定顺序，可加子节点，不可改顺序、不可去节
2. **PDCA 标注**（§2）：每节标注 `P` / `D` / `C` / `A` 图标
3. **"已落地 vs 未来" 严格分离**（§3）：整篇不写未来规划 / Next Step / Roadmap / 画饼（月报底部"下一步计划"为唯一例外）

> **判断示例**
>
> - "帮我做项目 X 的汇报 PPT" → 走 A + B（项目汇报）
> - "做这个月的月报" → 走 A + B（月报）
> - "把这次展会参观做成汇报 PPT" → **只走 A**（不强制 Agenda/PDCA，可自由组织章节，可写"后续建议/Next Step"）
> - "做个培训分享 PPT" → **只走 A**
> - "用 Agenda + PDCA 给我做这次外出参观汇报" → 走 A + B（用户明确要求）

非项目汇报场景（展会汇报、培训、分享、综述等）**鼓励**按内容自然组织章节，并且**允许写后续建议 / 行动项 / Next Step** —— §3 的"不写未来"约束此时不生效。

## 1. 固定 Agenda 顺序（不可改、不可去节）

> ⚠️ **本节属于 B 类规则，仅"项目汇报 / 月报"或用户明确要求时生效。** 展会汇报、培训分享等场景跳过本节，按内容自然组织章节。

| #   | 章节               | 英文(固定)   | PDCA         | 这一节要回答什么                                                 |
| --- | ------------------ | ------------ | ------------ | ---------------------------------------------------------------- |
| 01  | 项目背景           | `BACKGROUND` | `[PLAN]`     | 为什么做、痛点是什么                                             |
| 02  | 项目目标           | `TARGET`     | `[PLAN]`     | 目标是什么、要解决什么                                           |
| 03  | 项目范围           | `SCOPE`      | `[PLAN]`     | 边界，包含什么、不包含什么                                       |
| 04  | 项目应用场景       | `USE CASES`  | `[PLAN]`     | 在哪些场景落地                                                   |
| 05  | 项目节点           | `MILESTONE`  | `[DO]`       | 时间线、关键 milestone                                           |
| 06  | 项目行动与成果     | `OUTCOMES`   | `[DO/CHECK]` | **已完成的工作**和**已拿到的结果** —— 禁止写"将要 / 计划 / 未来" |
| 07  | 项目团队           | `TEAM`       | —            | 谁参与、分工                                                     |
| 08  | 持续优化与经验沉淀 | `LESSONS`    | `[ACT]`      | **上线后已落地的迭代**和经验沉淀 —— 禁止写未来 roadmap           |

**英文标题为强制项,不可改写**(例如不能把 `TARGET` 改成 `OBJECTIVES`、`MILESTONE` 改成 `TIMELINE`、`USE CASES` 改成 `SCENARIOS`)。
统一英文有两个作用:(1) 跨项目汇报视觉一致;(2) 听众扫一眼英文就知道这是 Agenda 第几节,不需要重新理解。

子节点英文标题自由(例如 06.2 可以叫 `GATEWAY` 或 `MIGRATION`),只要从属于父节即可。

可加子节点（如 `06.1 报表上线明细`、`06.2 用户反馈`），但：

- 不可改 8 节顺序
- 不可删除任意一节
- 子节点必须从属于父节，不可独立成章

## 2. PDCA 思维 — 每节都要标注阶段

> ⚠️ **本节属于 B 类规则，仅"项目汇报 / 月报"或用户明确要求时生效。** 其他场景不加 PDCA 图标。

**位置固定**：PDCA 标签放在**正文区右上角**，做成一个小图标（圆形 / 圆角方块 + 单字母），不要写进红 banner 的副标题里。
标签值固定为 `P` / `D` / `C` / `A`，与上表对应。第 07 节"项目团队"不带 PDCA 标签。

### 图标规范

- 形状：直径 ~24–32px 的圆形或圆角方块（PPT 中约 0.3–0.4 英寸）
- 位置：正文区右上角，距离顶部 banner 下沿 ~16px，距离右边 ~24px
- 颜色：
  - `P` / `D` / `C` / `A` 四种统一用 Milwaukee Red `#DB011C` 填充 + 白色字母
  - 或：白底 + 红描边 + 红字母（与该页其他红色元素抢眼时使用）
- 字号：单字母居中，约 12–14pt Bold
- 同一节多页时每页都要标，标签一致

### 示意

```
┌─────────────────────────────────┐
│ [Milwaukee 红色 banner: 项目背景]      │
├─────────────────────────────────┤
│                              ●P  │   ← PDCA 小图标
│  报表分散管理已成为扩展瓶颈           │
│  ...正文 lead 与论证...              │
└─────────────────────────────────┘
```

### 已落地图标资源

本 skill 已包含设计好的 PDCA 圆环图标 (`assets/pdca/{P|D|C|A}_icon.svg`):

- 4 段圆弧组成完整循环, **当前阶段那一段红色 `#DB011C`, 其余三段灰色 `#E5E5E5`**
- 中心字母红色 Arial Black, 字号 32pt
- 圆环造型表达 PDCA 是循环过程, 比单纯红圆 + 白字信息量大
- 直接复用即可, 不必另画

## 3. "已落地 vs 未来" 严格分离 — 强制规则

> ⚠️ **本节属于 B 类规则，仅"项目汇报 / 月报"或用户明确要求时生效。** 展会汇报、培训分享等场景**允许**写后续建议 / 行动项 / Next Step。

这是用户明确的强制规则（项目汇报场景下），违反会被驳回。

- **第 06 节"项目行动与成果"** = 已完成的工作 + 已拿到的结果
- **第 08 节"持续优化与经验沉淀"** = 上线后已落地的迭代和经验
- **整篇汇报不写**：未来规划 / Next Step / Roadmap / "下一步要做什么" / 画饼

### 自检清单（写完每节扫一遍）

- [ ] 这一节有没有出现"将要"、"计划"、"未来"、"下一步"、"roadmap"？
- [ ] 第 06 节里所有动词是不是都是已完成态？
- [ ] 第 08 节里所有迭代是不是**上线后已经做过**的？

任一项答"否"，回去改。

### 月报例外（仅 §5 Page 3 适用）

月报底部允许出现"下一步计划"，但必须满足：

- 已经确定、有 owner、有时间节点的动作
- 不是空泛 roadmap、不是"探索方向"、不是"考虑做"

不满足上述条件就不要写。

## 4. 标题三层结构

中文工作汇报习惯目录式标题（"项目背景"、"项目节点"），但目录式标题不承担论证。采用**话题主标题 + action title 副标题 + 正文 lead 强化**的三层结构。

### 转换示例

| 主标题（话题）            | 副标题（action title）         | 正文 lead（章节大标题）                       |
| ------------------------- | ------------------------------ | --------------------------------------------- |
| `项目背景 / BACKGROUND`   | `报表分散已成为扩展瓶颈`       | `报表分散管理已成为扩展瓶颈`                  |
| `项目目标 / TARGET`       | `项目按目标全面交付`           | `项目按目标全面交付`                          |
| `关键数据 / KEY OUTCOMES` | `20+ 报表,9 个月稳定运行`      | `项目按目标全面交付`                          |
| `项目节点 / MILESTONE`    | `软件如期,硬件延后 1.5–2 个月` | `软件如期上线,硬件因 RFQ 流程延后 1.5–2 个月` |

### 写法约束

- 主标题：≤ 14 中文 / 28 英文字符，禁止换行；优先中英双语（`项目背景 / BACKGROUND`）
- 副标题：≤ 40 字符，必须是 action title（给结论或动作，不是话题）
- 正文 lead：重申结论 + 加数据支撑

### 验收

只读副标题序列，能不能拼出完整故事？不能就回去改副标题。

## 5. 页面布局规则

模板文件：`assets/template.pptx`（顶部红 banner + 左上 logo + 底部页脚）。

### Page 1 — Cover（封面）

- 项目级标题 + 副标题（不是章节标题）
- 不出现任何章节内容
- 汇报人 / 团队 / 日期 一行说明

### Page 2 — Body（正文页 · 默认布局）

只允许出现这几个元素：

- **顶部红色 banner**
- **左上**（banner 内）：Milwaukee logo
- **右上**（banner 内）：本页主标题 + 副标题
- **正文区右上角**：PDCA 小图标（见 §2）
- **正文区中部**：正文内容
- 底部模板自带页脚

### Page 3 — Monthly Body（月报正文页）

格式与 Page 2 一致，**额外允许**底部追加两块：

- **本月一句话总结**（必填）：一句话给月度结论
- **下一步计划**（若有）：受 §3 月报例外约束 —— 必须是已确定、有 owner、有时间的动作

### Before / After 布局（用于 §06 成果展示）

- 左 BEFORE / 右 AFTER 对置
- 中间放转换指示（箭头 / "→"）
- AFTER 侧用 Milwaukee Red `#DB011C` 做强调锚点
- 不要给 BEFORE 加红色 —— 红色必须只标 AFTER

## 6. 设计系统

### 6.1 不可变的品牌锚点（硬约束）

Milwaukee 品牌的硬约束**只有三项,仅限顶部 banner 区域**:

1. **顶部红色 banner** —— Milwaukee Red `#DB011C`,使用模板原样,不可改色不可缩减高度
2. **左上 Milwaukee logo** —— 白色版本,位于 banner 左上
3. **banner 内文字白色** —— 主标题 + 副标题(中英双语),不可改成其他颜色

**banner 以下的正文区颜色完全自由发挥**,只需遵循下面 §6.2 的色彩适配原则。

### 6.2 正文区色彩适配原则

正文用色必须与 banner 的 Milwaukee Red `#DB011C` 视觉协调,不能撞色或显得割裂。

#### 推荐色板(与 #DB011C 协调)

| 类别                   | 示例 hex                                                                           | 用途                                                       |
| ---------------------- | ---------------------------------------------------------------------------------- | ---------------------------------------------------------- |
| **暖中性灰**           | `#1A1A1A` 深灰 / `#6B6B6B` 中灰 / `#E5E5E5` 浅灰 / `#FFFFFF` 白 / `#FAF7F2` 暖奶白 | 主文 / 副文 / 分割线 / 背景                                |
| **品牌红家族**(类似色) | `#DB011C` 主红 / `#8B1E2C` 深酒红 / `#5F1417` 暗红 / `#FCE6E8` 浅红粉              | 关键锚点 / 二级强调 / 浅色底色                             |
| **深冷对比色**         | `#1A2B5C` 深海军蓝 / `#2D3B4E` 深石板灰 / `#1F3A2E` 深松绿                         | 用于"流程 / 数据 / 系统 / 未来"等冷概念,作为红的对比但不撞 |
| **工业暖辅助色**(慎用) | `#C9701F` 深橙 / `#8A5A3C` 焦棕                                                    | 工业机械感强,做次要分类时偶尔用                            |

#### 撞色判断方法

把候选色和 `#DB011C` 并排放,问自己:

1. **明度差够吗?** 红是中明度,候选色应明显更暗(深冷色)或更亮(浅灰白)。中明度的蓝/紫/绿一定撞。
2. **是不是暖系或暗冷?** 暖红 + 暖灰 = 协调;暖红 + 中明度冷色 = 撞。冷色要够暗才能上。
3. **灰度复印后还能看出层级吗?** 看不出 = 文字读不清,换色。

任一项不满足,换色。

#### 必须避免

- ❌ **中明度纯蓝**(`#2B5BB5`、`#1E90FF` 等) —— 与红明度接近 + 色相对立 = 撞色
- ❌ **亮绿、亮紫、亮粉、青绿** —— 与暖红冲突,显得轻浮
- ❌ **渐变填充 / 阴影 / 发光** —— Milwaukee 工业品牌一向 flat,这些效果会破坏质感
- ❌ **三种或以上的强色相** —— 整页颜色超过两个强色相就乱;保持"灰阶 + 1~2 个强色"

### 6.3 红色用量建议(推荐,非硬规则)

- **红色总面积建议 ≤ 25%(含 banner)** —— banner 自带的红已占约 15~17%,正文红色再用 8~10% 就够
- **banner 以外的红色仅作视觉锚点**:关键数字、AFTER 强调、milestone、章节起始
- **不要在正文区出现大面积红色色块** —— 会和 banner 抢戏,降低 banner 的锚定作用
- **没有理由的红色装饰条 / 红色边框 / 红色高亮框**建议去掉 —— 红色应承担信息而非装饰

### 6.4 字体阶梯

同一页最大字号 / 最小字号 ≥ 3:1,制造层级。

### 6.5 实战色板(已落地验证)

下表从 PowerBI Server 上线项目沉淀,**给 §6.2 "推荐色板" 一个具体的实施起点**。直接复用即可,不再调色。

| 角色              | hex       | 主要用途                                                               |
| ----------------- | --------- | ---------------------------------------------------------------------- |
| **Milwaukee Red** | `#DB011C` | 唯一品牌色:banner、关键数字、AFTER 强调、PDCA 红段、章节锚点条         |
| **深灰文字**      | `#1A1A1A` | 正文 lead、卡片主标题、icon row 标题、深灰大数字(与红字并列时)         |
| **中灰副信息**    | `#6B6B6B` | 描述句、副信息、注脚、来源标识、占位框元信息                           |
| **次重要灰**      | `#333333` | P1 优先级徽章底、次级标识                                              |
| **轻次要灰**      | `#777777` | P2 优先级徽章底、Cover 字段标签、tagline                               |
| **细分割线**      | `#E5E5E5` | 卡片描边、分割线、时间线主线                                           |
| **占位底灰**      | `#F5F5F5` | 图片占位框背景、BEFORE 卡片背景、灰阶强调底                            |
| **占位边框**      | `#D0D0D0` | 图片占位框描边、内嵌截图框                                             |
| **深海军蓝**      | `#1A2B5C` | 冷色辅助:经验沉淀小标、安全/合规标识、"流程 / 数据"概念,与红的暖色对比 |
| **白**            | `#FFFFFF` | 卡片底、内嵌截图框底、banner 内文字、AFTER 卡片底                      |

整套色板可简化记忆为 **"红 + 黑(深灰) + 白 + 4 档灰阶 + 1 个冷色辅助"**。
绝大多数页面只需要 `#DB011C` / `#1A1A1A` / `#6B6B6B` / `#E5E5E5` / `#FFFFFF` 这五个色就够。

### 6.6 三个高频组合(实战沉淀的视觉模式)

整套 PPT 反复使用这三种组合,形成"重复"原则下的视觉特征:

1. **正文 lead 一致格式**
   红色锚点条 `#DB011C` (0.25" × 0.06") + 紧贴 16pt 加粗 `#1A1A1A` 文字。
   每页内容区起首必有,作为"故事开始"的视觉信号。

2. **大数字 stat callout**
   60–80pt 数字(`#DB011C` 或 `#1A1A1A`) + 红色短线 (0.5" × 0.04") + 14pt 加粗 `#1A1A1A` label + 11pt `#6B6B6B` 描述。

3. **icon row**(替代 4+ 个 bullet)
   圆形数字标号 (0.5" 红色或深灰) + 14pt 加粗 `#1A1A1A` 标题 + 11pt `#6B6B6B` 一句话描述。
   首项常用红圆吸引视线,其余用深灰圆。

### 6.7 红色面积分配(实战量化)

§6.3 的 "≤ 25%" 是上限,实战中按页面类型分档更可控:

| 页面类型                    | banner 外红色 | 实战做法                                                                 |
| --------------------------- | ------------- | ------------------------------------------------------------------------ |
| Cover / Closing             | ≤ 8%          | 模板自带全红 Cover 已足,不再叠加                                         |
| 标准正文页                  | ≤ 5%          | 1 个关键数字 + 锚点条 + PDCA 图标                                        |
| BEFORE / AFTER 对比页       | ≤ 5%          | AFTER 顶条 + 描边 + 箭头                                                 |
| 多 stat 并列页(3+ 个大数字) | ≤ 5%          | **只让 1 个用红,其余 2-3 个用 `#1A1A1A` 深灰** ── 通过大小造对比而非颜色 |

### 6.8 实战禁区(在 §6.2 "必须避免" 之上的补充)

- ❌ Cover 自创深黑底 ── 模板 Cover 已是全红,不要再叠黑底
- ❌ 用 `#DB011C` 作大段背景填充 ── 红色仅用作锚点,不作背景
- ❌ 三个大数字全部用红 ── 一定要让其中 2 个改深灰,造大小对比而非色相对比

### 6.9 红色的语义 —— 只标"已完成"或"最危险"

§6.3 / §6.7 / §6.8 规定了红色用**多少**、放**哪类元素**;这一节规定红色承载什么**意义**。一句话:

> **红色 = "这件已搞定" 或 "这件最危险",二选一。除此之外全交给黑白灰。**

两种语义对应两类页面:

| 页面性质                | 红色标注对象             | 实战做法                                                                 |
| ----------------------- | ------------------------ | ------------------------------------------------------------------------ |
| **成果页 / 总览页**(正) | 已完成、最该被记住的成果 | KPI 数字中染红"已落地"的那个;进度行里"已落地"用红、"推进中 / 规划中"用黑 |
| **问题页 / 现状页**(反) | **风险最高的那一个**痛点 | 多痛点并列时只给 1 个点红(红标题 / 左侧红竖条),其余中性灰                |

**问题页判据**(哪个痛点配得上那一点红):能造成**业务中断 / 单点故障 / 不可逆损失**的 > 仅影响效率的。
例:升级前 4 个痛点中,"个人化开发 → 人走刷新断、单点故障"是唯一的业务中断风险,只点它;"报表分散 / 无标准 / 不沉淀"是效率问题,保持灰。

与 §6.7 呼应:成果页"只让 1 个数字用红"和问题页"只点 1 个痛点用红",是同一条纪律的两面 —— **红色永远在做"选择",不做"平铺"**。

补 §6.8 禁区:

- ❌ 问题页把所有痛点都染红 ── 等于没有重点,满屏告警反而钝化,读者抓不到"哪个最该急"

## 7. 内容硬约束

### 每节一个核心信息

- 一节只回答一个问题，禁止多议题混搭
- 内容超出时拆成两节，不要塞满
- bullet list 是最后选择，能用大数字 / 对比 / 时间线 / 流程图就用

### 禁用元素

- ❌ **Emoji 占位符**(📷 📊 📈) —— 改用真实数据或简短文字
- ❌ **两栏 bullet list 堆砌** —— AI slop 标志
- ❌ **同一节 4+ 个 bullet 段落** —— 改用 card 网格 / 表格 / 图表
- ❌ **SmartArt 风格的方框+箭头流程** —— 改用极简数字标号 + 文字
- ❌ **空话套话开头**("在领导关怀下"、"经过不懈努力") —— 直接给结论
- ❌ **画饼词汇**("打造"、"赋能"、"未来可期"、"展望") —— 全部禁用
- ❌ **渐变 / 阴影 / 圆角发光** —— Milwaukee 工业品牌保持 flat

## 9. 不要做的事

- 不要写"下一步"、"未来计划"、"Roadmap"(月报的"下一步计划"为唯一例外,且必须是已确定动作)
- 不要改 Agenda 8 节的顺序
- 不要删除 Agenda 中的任何一节(无内容也要保留并标注"本期无更新")
- 不要把 action title 放在主标题位置 —— 长结论会溢出,且不符合中文汇报体例
- 不要让副标题和正文 lead 完全相同
- 不要用目录式标题承担论证 —— "项目背景"只是话题,结论必须由副标题给出
- 不要写空话套话 / 画饼词汇
- 不要在同一节混多个议题
- 不要给 BEFORE 加红色 —— 红色锚点专属 AFTER
- 不要在正文区用与品牌红撞色的强色相(中明度纯蓝、亮绿、亮紫、亮粉)
- 不要用渐变填充、阴影、发光、毛玻璃等效果 —— Milwaukee 是工业 flat 风
- 不要改顶部 banner 的红色、logo、白色文字 —— 这三项是品牌硬约束
