# 1. sediment 拆成"顾问 + 账本"两个 skill

日期:2026-07-07
状态:已采纳

## 背景

想把参考 skill(pskoett/self-improvement)的两个设计抄进 sediment:量化晋升门槛(复发≥3 × 跨2任务 × 30天窗口才写 CLAUDE.md)和 Pattern-Key 去重,并加一个 commit 后自动触发的 hook。

难点:这两个机制都是**有状态**的——要跨会话数"第3次复发""30天窗口",必须有个持久存储。而当时的 sediment 是**无状态顾问**:只看当前会话、只给建议、其硬约束明写"不要擅自创建或修改任何文件"。把有状态机制直接塞进去,会毁掉它"只诊断不落库"的洁癖。

## 决策

**拆成两个 skill,不改 sediment。**

- `sediment`:保持原样——手动调用的复盘顾问,无状态、只建议、不写文件。
- `sediment-basin`(新):承接落库脏活——把经验持久堆进每项目 `.sediment/ledger.md`,按 Pattern-Key 去重、累加复发,达成岩阈值时冒泡请人转正、写 CLAUDE.md 或抽成 skill。

配套决策:

- **任务单位取 commit**:"跨2任务"里的"任务"用 commit SHA 计数。hook 挂在 commit 后,天然知道当前 SHA,无需额外机制。
- **hook 用 Claude Code PostToolUse(全局)**:模型在场,能当场判断信号、打 Pattern-Key。纯 git 钩子没有模型,做不到。
- **转正人守闸**:达阈值只冒泡,人点头才写 CLAUDE.md;另留"手动放行"旁路,不必等凑满阈值。
- **门槛可调**:3 / 2 / 30 是默认值,存在账本 config 里,单人场景嫌严可调松。

## 后果

**好处**

- sediment 的顾问洁癖不受污染;两者职责清晰,同构于参考的 self-improvement / self-healing 分工。
- 有状态的账本独立演进,阈值、去重逻辑改动不牵连顾问。

**代价 / 风险**

- 将来读到两个名字相近的 skill 会犯嘀咕"为啥不合成一个"——本 ADR 就是答案。
- **Pattern-Key 漂移**:去重押在"先搜后铸"这道人为纪律上,漏搜就断链;靠月度复盘的"近义键合并体检"兜底。
- **单人场景阈值可能永不触发**:一个人一个项目里 30天内同一模式复发3次跨2commit未必凑得齐;靠"手动放行"+"可调门槛"缓解。

## 备选方案

- **直接改 sediment 成全自动累积器**:放弃"不写文件"约束,达阈值静默改 CLAUDE.md。否决——毁掉顾问洁癖,且静默改项目指令太激进。
- **合成一个 skill 但内部分模式**:顾问模式 + 账本模式同处一个 SKILL.md。否决——一个 skill 同时"无状态只建议"和"有状态改文件",描述与约束自相矛盾,触发时机也难写清。
